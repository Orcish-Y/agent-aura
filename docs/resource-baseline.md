# Agent Aura WPF 原型：变更后资源基线

## 结论

2026-07-13 在 Windows 11 实机上完成了一次可重复的 30 分钟稳态采集：5 分钟持续执行 Window Pin State 与三个 Agent Message Item 的悬停交接，随后静置观察 25 分钟。采集得到全部 360 个预期稳态样本，最大间隔 5.099 秒，没有睡眠、崩溃或采集缺口。

框架依赖的 Agent Aura 包只有 **0.220 MiB**。机器级 .NET 10 Core 与 Windows Desktop 共享运行时为 **170.367 MiB**，是多个 .NET 应用可共用的先决条件，不属于 Agent Aura 包。

WPF 原型的内存成本仍然偏高。全场景私有内存 P95 为 **234.289 MiB**，工作集 P95 为 **308.535 MiB**；后者是当时驻留在物理内存中的页面，其中一部分可被 Windows 回收，不能解释成 Agent Aura 永久独占了同等物理内存。25 分钟静置期没有出现句柄或私有内存单调增长，但结果仍明显超过既有的 100 MiB 空闲工作集门槛。

因此，本结果不否定 WPF，也不触发 Tauri 对比。WPF 已通过核心能力验证；这些数字用于确定 WPF 的优化目标和回归护栏。只有必需的核心能力无法在 .NET/WPF 中实现时，才重新考虑 Tauri。

## 测量环境与场景

- Windows 11 家庭版 `10.0.26200`，64 位。
- Surface Pro 9，Intel Core i7-1255U，12 个逻辑处理器，16 GiB 内存。
- `Microsoft.NETCore.App` 与 `Microsoft.WindowsDesktop.App` 10.0.9。
- 源码基点 `078e993b368d63bf432d7c6874c9ebcbf2b05d8f`，环境快照同时保留了脏工作树状态。
- 发布框架依赖 `win-x64` 包，然后复制到 Windows 本地临时目录运行；不从 WSL 的 UNC 共享路径直接启动。
- 做 5 次全新进程启动，每次之间等待 10 秒。
- 预热 15 秒；交互 5 分钟，每 5 秒完成一次指针移出、Pinned Header 隐藏、重新进入以及三个 Agent Message Item 的直接悬停交接；随后静置 25 分钟。
- 采集期间阻止系统自动睡眠；脚本退出后恢复默认电源行为、原鼠标位置和原窗口状态。

## 结果

| 指标 | 结果 | 解释 |
| --- | ---: | --- |
| 框架依赖包 | 0.220 MiB | 不包含机器级 .NET 运行时。 |
| 共享 .NET 10 运行时 | 170.367 MiB | 单独记录的机器级先决条件。 |
| 5 次启动 | 3.561 / 2.411 / 2.267 / 2.451 / 2.495 秒 | 从用户启动器开始，到无任务栏按钮的观察窗口可由 UI Automation 访问。中位数 2.451 秒。 |
| 连续性 | 360 / 360 个稳态样本 | 60 个交互样本、300 个静置样本；最大间隔 5.099 秒。 |
| 全场景私有内存 | 中位 200.633 MiB；P95 234.289 MiB；最大 239.695 MiB | 进程自己的私有提交内存，不等同于全部驻留物理内存。 |
| 全场景工作集 | 中位 279.555 MiB；P95 308.535 MiB；最大 314.422 MiB | 当前驻留页，可在内存压力下被部分回收。 |
| 全场景 CPU | 平均 0.359%；P95 1.814%；最大 6.308% | 已除以 12 个逻辑处理器。 |
| 交互期 | 私有内存 P95 238.441 MiB；工作集 P95 312.582 MiB；CPU 平均 1.666% | 持续执行 Header 与条目悬停动画。 |
| 静置期 | 私有内存 P95 221.562 MiB；工作集 P95 294.156 MiB；CPU 平均 0.098% | 保持窗口可见并置顶，不注入 Codex 事件。 |

静置期首尾单样本变化为私有内存 **+1.805 MiB**、工作集 **−1.882 MiB**、句柄 **+2**。首个与最后一个 5 分钟窗口的中位数变化为私有内存 **+17.230 MiB**、工作集 **+10.992 MiB**、句柄 **−9**。五个连续 5 分钟窗口的私有内存中位数依次为 204.332、169.316、195.305、195.305、221.562 MiB，说明过程包含回收和重新提交，并非持续单调上涨。

## 门槛与回归预算

### 轻量性能目标

既有轻量原型门槛保持不变，不因本次结果而放宽：

- 全新进程启动中位数不超过 1.5 秒；本次 WPF 为 2.451 秒，未达到。
- 静置工作集不超过 100 MiB；本次 WPF 静置 P95 为 294.156 MiB，未达到。
- 静置 CPU 平均不超过 1%；本次 WPF 为 0.098%，达到。
- 发布包不超过 200 MiB且必须把共享运行时单列；本次 WPF 包为 0.220 MiB，达到。

未达到这些目标会触发 WPF 内部的测量、分析和优化，不会自动触发另一套 UI 技术原型。只有出现无法在 .NET/WPF 中实现的必需核心能力时，才重新考虑 Tauri。

### 当前 WPF 基线的回归护栏

后续 WPF 原型变更使用以下护栏判断是否相对本基线发生物质退化：

- 必须得到 360 / 360 个稳态样本，最大采样间隙不超过 10 秒，且无崩溃或卡死。
- 框架依赖包不超过 1 MiB，共享运行时继续单列。
- 5 次全新进程启动中位数不超过 3.0 秒。
- 全场景私有内存 P95 不超过 260 MiB，工作集 P95 不超过 340 MiB。
- 全场景 CPU 平均不超过 1%，P95 不超过 3%。
- 静置首末 5 分钟窗口的私有内存中位数增加不超过 32 MiB，句柄中位数增加不超过 32；不得跨连续 5 分钟窗口持续单调增长。

任一数值护栏超限即标记为回归并在相同环境复跑；第二次仍超限时，必须启动 WPF 性能调查。崩溃、卡死、样本不完整，或私有内存/句柄持续单调增长不等待第二次复跑，直接触发调查。上述护栏只是相对当前 WPF 基线的回归线，不取代更严格的轻量性能目标。
